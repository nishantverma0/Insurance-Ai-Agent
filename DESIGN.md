# Design Doc — Carrier Form Automation Agent

## 1. Perception: structured DOM extraction, not vision
The agent extracts a **structured field list** from the live DOM via `page.evaluate` (see `src/perceive.ts`) rather than screenshots + a vision model: for each visible `input`/`select`/`textarea`/radio-group it captures label (via `label[for]`, wrapping `<label>`, or nearest preceding label/heading), type, placeholder, inferred helper text, `required`, current value, and options. Radio groups are collapsed into one logical field. This is ~10-50x cheaper and faster per page than screenshots, and label/type/options are exactly the evidence an LLM needs for semantic mapping — no pixel reasoning is required. Repeating rows are detected structurally (sibling containers with identical input-type signatures next to an "add"-labeled control), and visible validation errors are extracted via generic class/role heuristics (`*error*`, `*invalid*`, `role=alert`). Every element is tagged with an opaque `data-agent-ref` handle at perception time, so downstream code never touches a page's real ids/names/selectors.

## 2. Mapping: heuristics first, one batched LLM call per page for the rest
`src/map.ts` first tries deterministic matching (normalized exact match, then token-overlap ≥0.75) between each field's label and the answermap's key names — cheap, and safe because the bar is high. Everything the heuristics can't confidently resolve is batched into a **single LLM call per visible page** (`src/llm.ts`), which returns a JSON array of `{refId, answerKey, confidence, reason}`. Select/radio option text is resolved separately via a small matcher (exact → normalized → acronym-expansion → token-overlap fuzzy), so "LLC" correctly resolves to "Limited Liability Company" without the LLM needing to see or guess exact option casing. Repeating sections are mapped **once** against a single template row, then the field↔item-key mapping is reused positionally across all rows — so N rows cost one mapping call, not N.

## 3. Missing/ambiguous-answer policy
Three tiers, in order:
1. **Confident match** (LLM/heuristic confidence ≥ 0.4): value is filled/selected/checked. Matches in [0.4, 0.6) are still filled but *also* added to the flagged-for-review list, since a low-confidence fill is exactly the kind of thing a human reviewer should double-check before submission.
2. **No confident match, required field**: flagged for human review (not filled, not guessed) — except one narrow, explicitly logged exception: a required boilerplate certification/acknowledgement checkbox (e.g. "I certify this is accurate") with no answermap key is auto-checked, because it attests to the accuracy of data the agent itself just entered, not a new fact the agent would have to invent.
3. **No confident match, optional field**: skipped and left at its default.
All three are logged with full reasoning in the decision log; the human-review list is written to `<form>.confirmation.json` (`fieldsFlagged`).

## 4. Cost & latency
Per form (3–4 pages): roughly **1 LLM call per page** for ambiguous top-level fields (most fields are resolved by the free heuristic pass), plus **1 extra call per repeating section** (paid once regardless of row count). For these sample forms that's ~4–5 calls total, each a few hundred input tokens (field list + answer-key catalog) and well under 500 output tokens (JSON only). At scale, the highest-leverage optimization is **caching the resolved field→answer-key map per carrier+form-version** (keyed by a hash of the perceived field signature) after the first successful run, and only re-invoking the LLM when perception detects the signature has changed — turning steady-state cost toward zero LLM calls per submission.

## 5. Top failure modes
1. **Non-standard input surfaces**: canvas-rendered forms, custom JS "select" widgets built from `div`/`li` instead of `<select>`, shadow-DOM components, or iframed third-party form builders won't appear in a plain `input/select/textarea` DOM query. Mitigation: extend perception to pierce shadow roots (`el.shadowRoot`) and add an accessibility-tree pass (`page.accessibility.snapshot()`) as a fallback so ARIA-role-based custom widgets are still discovered; canvas-rendered forms would need a vision-model fallback, which is the main case where screenshots earn their cost.
2. **Infinite-scroll / async-loaded dropdown options and multi-step client-side validation with non-deterministic timing**: an option list populated by a debounced async search (e.g. "type to search carrier name"), or a Continue button that re-validates via a network call before revealing the next step, can race the agent's fixed `waitForTimeout`. Mitigation: replace fixed timeouts with `waitForResponse`/`waitForSelector` on the specific DOM delta expected, and for searchable dropdowns, type into the field and wait for the option list to stabilize before matching.
3. **Segmented native inputs (`<input type="date">`, etc.) rejecting keystroke-simulated text**: native date inputs render separate month/day/year segments rather than accepting a typed string; simulating keystrokes for an ISO date (including the `-` separators) distributes characters across the wrong segments and silently corrupts the value instead of erroring — observed in testing, where typing `"2026-08-01"` into a date input landed as `"60801-02-20"`. Mitigation (implemented): detect `field.kind === "date"` and set the value via Playwright's `locator.fill()`, which sets segmented date inputs directly rather than simulating keystrokes; kept keystroke-typing for text/combobox/autocomplete fields that rely on real input events to trigger masks or search-as-you-type.

(CAPTCHAs, logins, and anti-bot measures are out of scope per the brief; in production they'd require a human-in-the-loop hand-off rather than automation.)
